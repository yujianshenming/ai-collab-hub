# 个人工作台 V3.3「文件总线可靠性 + 任务源头闭环」规格书（Round 8）

> 产品经理产出 · 2026-06-11
> 前置依赖：V3.2 任务源头已交付（commit 39fd54c 及之前），V3.2 六条验收由测试工程师并行执行
> 改造对象：`personal-workbench/main.js`、`renderer.js`、`preload.js`、`index.html`、`style.css`

---

## 0. 背景与目标

V3.1/V3.2 把「下载捕获 → 托盘 → 上传注入」和「待做任务.txt 单向导入」搭起来了，但回归清单（`personal-workbench/regression-checklist.md` 附表）登记了 6 个未修复缺陷，其中 **#2 直接威胁文件总线核心价值**（上传拦截可能从未生效），**#5 会污染任务产物**（任意文本文件被误改名 dialogue.json 并误推流水线）。

同时 V3.2 只做了 txt → 工作台的单向导入，用户在工作台里更新任务状态后，桌面的待做任务.txt 又过期了，两边还是双维护。

**V3.3 目标：①已知缺陷清零，把文件总线做可靠；②任务状态写回待做任务.txt，完成任务源头双向闭环。**

## 1. 功能一：已知缺陷清零（按优先级实施）

> 编号沿用 `regression-checklist.md` 附表。#1（findTabByUrlPart）已由 commit 3db... 修复，不在本轮范围。

### 1.1 缺陷 #2（P1）：上传拦截机制重做

- **现状**：`renderer.js` `setupWebviewUploadInterceptor` 监听 `select-file-dialog`，该事件不在 Electron 36 官方 webview 事件列表中，上传拦截/评估页自动注入大概率从未触发，一直走系统选择器。
- **要求**：
  1. 先真机验证现状（活动任务期间任意网页点上传，确认浮层是否弹出），把结论写进回报。
  2. 重做拦截机制。候选技术路径（架构师可自行评估，但必须满足行为验收）：main 进程对 webview 的 webContents attach CDP debugger，`Page.setInterceptFileChooserDialog(true)` 拦截 + `DOM.setFileInputFiles` 注入所选绝对路径。
  3. 行为与 V3.1 规格 §3 完全一致：活动任务弹工作台浮层（多选/改用系统选择器/ESC 取消）；评估页流水线步骤保持自动注入不弹层；无活动任务不拦截。
  4. 若拦截开启期间发生异常，必须降级回系统选择器，不允许上传按钮点了没反应。

### 1.2 缺陷 #5（P2）：下载分类误判修复

- **现状**：`main.js` 下载分类对任意 .json 命中 chat 类型即改名 `dialogue.json` 并推进流水线；.pdf 同理误判 eval_report。
- **要求**：
  1. 收紧分类条件：**仅当下载来源 URL 属于已登记的测试页/评估页域名**（现有 wl363eval 判断逻辑泛化成常量表）时才走 dialogue/eval_report 特殊归档与流水线推进。
  2. 其它一切下载走 V3.1 通用规则：保留原名落任务文件夹，重名追加 ` (2)`。
  3. 流水线步骤推进必须与特殊归档绑定（只有真 dialogue/report 才推 step）。

### 1.3 缺陷 #3（P2）：标签销毁资源泄漏

- **现状**：`renderer.js` 三处用 `DOMNodeRemovedFromDocument` 做清理（560/697/902 行附近），该突变事件 Chromium 127+ 已移除，删除标签后桌面应用轮询定时器、CLI 数据监听器、resize 监听器全部泄漏。
- **要求**：建立**统一的标签资源清理函数**（per-tab cleanup registry：tabId → cleanup[]），在删除标签的代码路径里显式调用；废弃全部 `DOMNodeRemovedFromDocument` 监听。删除 CLI 标签杀 pty 的既有逻辑（`tab:cleanup-resources`）并入同一出口。

### 1.4 缺陷 #4（P2）：非 web 标签点扩展按钮崩溃

- **现状**：非 web 标签激活时点顶栏扩展按钮，`extBody` 为 null 抛 TypeError。
- **要求**：guard + `showToast("当前标签页不支持扩展面板", "error")`，不崩、不静默。

### 1.5 缺陷 #6（P3）：Hermes 提示词换行

- `renderer.js` `runHermesPrompt` 中 `join("\\n")` 改为真实换行 `join("\n")`，注入 Hermes 输入框后是多行文本。

### 1.6 缺陷 #7（P3）：流水线 prepare 步骤跳空

- 点「执行」先置 `prepare`（1/5），任务文件夹创建成功后再推进 `testing`（2/5）；失败停在 prepare 并 toast 报错。把手进度环/状态栏芯片随动。

## 2. 功能二：任务状态写回待做任务.txt（闭环）

### 2.1 触发方式

- **只做手动触发，不做自动写回**（用户的 txt 是真实工作文件，防意外覆盖）。
- 任务中心页头「从待做任务导入」旁新增 ghost 按钮「写回待做任务」。

### 2.2 写回规则

- 匹配键沿用 V3.2：`学校 + 课程 + 任务类型` 三元组定位 txt 中的行。
- **只改写匹配行的：数量、状态、括号备注（由 subtasks 反向生成）、负责人、星期**；行首「学校《课程》任务类型」原样保留，未匹配的行（含无法解析行）一字不动。
- 状态反向映射：pending→`未完成`、unsubmitted→`未提交`、completed→`已完成`。
- 括号备注反向生成规则：subtasks 里有 done 的 → `（已完成任务x/y/z）`；有 unconfirmed 的 → 追加 `，任务n待确认`；全 pending 则不写括号。与 V3.2 解析器**互为逆运算**：写回后再导入必须全部落「无变化」组。
- 工作台有、txt 没有的任务：**默认不追加**，在预览中列为「仅工作台」分组供用户勾选追加（追加行按 §0 样例格式生成，追加到文件末尾）。

### 2.3 写回交互与安全

- 点击后弹**写回预览 dialog**（复用导入预览风格）：逐行显示 旧行→新行 diff；分组：变更 / 无变化（折叠）/ 仅工作台（默认不勾）。
- 确认写回前，main 进程先在同目录生成备份 `待做任务.txt.bak`（每次覆盖，保留最近一次）。
- 写文件 UTF-8（无 BOM 则不加，有 BOM 则保留），换行符保持原文件风格（CRLF/LF 检测后沿用）。
- 新 IPC `tasks:write-todo-file`：只允许写偏好中登记的那一个 txt 路径 + 同目录 `.bak`，preload 不暴露通用写文件能力。

## 3. 范围外（明确不做）

- 自动/定时写回（仅手动按钮）
- 企业微信抓取
- 拖拽文件拖进 webview 上传区（startDrag，继续顺延）
- GBK 转码（维持 V3.2 口径：提示转存 UTF-8）
- 跨任务全局文件库

## 4. 验收标准

1. **#2 上传拦截**：活动任务期间在任意网页（含评估页之外的页面）点上传 → 工作台浮层弹出，多选注入成功、注入后页面能读到文件；「改用系统选择器」与 ESC 取消正常；评估页流水线自动注入不弹层；无活动任务不拦截。拦截异常时降级系统选择器。
2. **#5 下载分类**：活动任务期间从非评估页下载任意 .json/.pdf/.txt → 保留原名落任务文件夹，流水线步骤不动；评估/测试页下载 → dialogue.json / eval_report 归档与推进不回归。
3. **#3 资源清理**：删除含桌面应用/CLI 的标签后，对应轮询定时器与监听器全部释放（可用计数日志或 devtools 验证），CLI pty 被杀。
4. **#4/#6/#7**：非 web 标签点扩展按钮出 toast 不崩；Hermes 注入文本为真实多行；执行任务从 1/5 prepare 起步。
5. **写回**：用 V3.2 规格 §0 的 8 行样例导入 → 工作台改状态/子任务 → 写回 → txt 仅目标行变化、备份生成、未解析行原样；**写回后立即再导入 → 全部落「无变化」**（逆运算自检）；「仅工作台」任务默认不写、勾选后正确追加。
6. **零回归**：`npm run check` 通过；V3.1 文件总线、V3.2 导入、分屏/终端/扩展/安全四项按回归清单抽测通过；新 IPC 仅能写偏好登记的单一 txt 路径与其 .bak。

### 提交要求

- 三个 commit：①缺陷 #2+#5（文件总线可靠性）②缺陷 #3+#4+#6+#7（稳定性杂项）③功能二（写回），各自过 `npm run check`。
- 缺陷 #2 的真机验证结论（修复前是否确实失效）写入回报。
- 写回的「行生成/行替换」逻辑写成无副作用纯函数并导出（如 `serializeTodoLine(task)` / `mergeTodoFile(text, tasks)`），便于测试工程师写用例。
- 完成后向产品经理回报：改动清单 + 六条验收自检；验收通过后按 GitHub 同步策略推送当前分支。
