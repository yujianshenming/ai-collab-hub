# CLAUDE.md (AI 协作与编码规范)

本文件定义了在该项目中 AI 编码助手（Antigravity 与 Codex）的行为准则、开发工作流与技能配置。

---

## 1. 核心行为准则 (Karpathy 行为指南)

本准则旨在减少大模型常见的编码错误，倾向于“谨慎与质量”而非“盲目速度”。

### 1.1 编码前思考 (Think Before Coding)
* **明确假设，避免盲目**：在执行前，先口头明确你的假设。如果不确定，直接向用户提问，不要靠猜测来做决策。
* **呈现权衡**：如果存在多种实现方案或存在歧义，将它们罗列并解释优缺点，不要默默挑选一个默认方案。
* **适时反驳**：如果有更简洁的实现方式或发现设计不合理，应向用户指出。
* **困惑时停下**：如果对上下文或需求感到困惑，立即暂停并寻求澄清。

### 1.2 简洁优先 (Simplicity First)
* **最小化实现**：只编写解决当前问题所必需的最少代码，不做任何投机性、面向未来的开发。
* **不制造多余抽象**：不要为仅使用一次的代码创建抽象，不要添加未被明确要求的“灵活性”或“配置项”。
* **避免无用纠结**：不要为不可能发生的场景做过度错误处理。
* **重构原则**：如果你写了 200 行，但发现 50 行就能搞定，请果断进行精简重写。
* **黄金法则**：时常问自己：“一位资深工程师会觉得这个设计过于复杂吗？”如果是，请立刻简化。

### 1.3 精准修改 (Surgical Changes)
* **触碰最小化**：仅修改为了达成任务必须修改的行。不要顺手重构、格式化或“改进”相邻的无关代码或注释。
* **尊重原有风格**：强制匹配项目已有的代码风格和命名规范，即使你个人或模型习惯不同。
* **只清理自己制造的混乱**：清理因你的改动而变得多余的 import、变量或函数；不要删除预先存在的死代码，除非被明确要求。如果发现无关死代码，只需向用户提及。

### 1.4 目标驱动执行 (Goal-Driven Execution)
* **将任务转化为可验证的目标**：
  * “添加验证” ➔ “为无效输入编写测试，然后编写代码让其通过”。
  * “修复 bug” ➔ “编写能够重现 bug 的测试，然后使其通过”。
  * “重构模块” ➔ “在重构前后确保所有单元测试和功能均能通过”。
* **任务计划书**：对于多步骤任务，先给出一个简短的计划：
  ```
  1. [步骤 A] ➔ 验证方法: [检查 A]
  2. [步骤 B] ➔ 验证方法: [检查 B]
  ```

## 2. Agent 技能配置 (Agent Skills)

本配置供 Matt Pocock 技能工具箱读取，用来对齐任务追踪和领域上下文。

### Issue tracker

本地 Markdown 任务看板。任务存放在 `tasks/` 目录下（`todo.md`、`active.md`、`done.md`）。详见 [issue-tracker.md](file:///e:/codes/docs/agents/issue-tracker.md)。

### Triage labels

通过在 `tasks/` 目录的不同文件间移动任务来管理状态（`todo` ➔ `active` ➔ `done`）。详见 [triage-labels.md](file:///e:/codes/docs/agents/triage-labels.md)。

### Domain docs

单上下文模式。项目领域词汇与协作规则由根目录的 `PROTOCOL.md` 以及 `decisions/` 目录定义。详见 [domain.md](file:///e:/codes/docs/agents/domain.md)。
