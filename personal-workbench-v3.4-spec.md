> 改造对象：`personal-workbench/main.js`、`renderer.js`、`preload.js`、`index.html`、`style.css`

## 0. 背景与目标

用户流程中最大的剩余人工黑洞：**Hermes 产出 → 公司平台逐字段誊写**。

每个任务 N 张卡片，每张卡要把「卡片名称 / 建议轮次 / 阶段描述 / 开场白 / 提示词」五个字段逐项从 Hermes 会话里选中复制 → 切到平台标签 → 找到对应输入框粘贴，另有任务描述、评价标准、测试人格等总任务级字段。一个 6 卡任务要做 30+ 次「选中-切页-定位-粘贴」。

**V3.4 目标：把「逐项誊写」压缩成「逐项点一下」。** 分两层：
- **P0 卡片舱**（本期必达）：Hermes 产出解析成结构化卡片包，任务舱里逐字段一键复制，流式引导
- **P1 平台注入勘测**（本期预研）：为下一步「点一下都省掉」摸清公司平台表单结构

## 1. 功能一（P0）：卡片包解析

### 1.1 产物来源约定

- 任务文件夹下约定文件 `cards.md`（用户把 Hermes 最终产出全文保存进去，或经文件总线直接落盘）。
- 托盘中识别到 `cards.md` 时自动解析；解析入口同时挂手动按钮（卡片舱头部「重新解析」）。

### 1.2 解析规则

Hermes 产出有稳定结构（能力训练卡片五项制）。解析为：

```
{
  taskMeta: { taskName?, taskDescription?, coverDescription? },
  cards: [ { name, rounds, stageDescription, opening, prompt } ],
  evaluation?: string,   // 评价标准整段原文
  testPersona?: string   // 测试人格整段原文
}
```

- 卡片边界锚点：`卡片名称`（或 `## 卡片` 级标题）出现即开新卡；五项字段按「卡片名称 / 建议轮次 / 阶段描述 / 开场白 / 提示词」标签行识别，标签后内容直到下一标签为止。
- 提示词识别代码块：```` ``` ```` 包裹的整块作为 prompt（保留内部换行与 markdown，复制时原样）。
- 评价标准 / 测试人格：按对应标题锚点截取整段原文，**不做内部拆分**（格式多变，拆错比不拆更糟）。
- 解析失败容错：无法归入任何字段的内容进 `unparsed` 列表，卡片舱单独展示原文，不阻塞已解析部分。
- **纯函数 `parseCardsDocument(text)` 导出**，便于测试用例覆盖。

## 2. 功能二（P0）：卡片舱 UI

### 2.1 布局

- 任务舱新增「卡片」区（与「任务产物」区平级，无 cards.md 时显示引导文案：「将 Hermes 产出保存为 cards.md 到任务文件夹」）。
- 逐卡折叠面板：标题行 = `卡片1 · {name}`，展开后五行字段，每行：字段名 + 内容预览（单行截断）+ **「复制」按钮**。
- 总任务级字段（任务描述/封面图描述/评价标准/测试人格）在卡片列表上方同样式排布。

### 2.2 交互细节

- 点复制 → 剪贴板写入该字段**完整原文** → 按钮变 ✓（2 秒恢复），该行标记「已复制」状态（淡色背景，持久到任务结束，便于知道填到哪了）。
- **流式模式**：卡片舱头部「逐项填写」开关。开启后高亮第一个未复制字段，每次复制自动滚动并高亮下一个未复制字段，把 30 次定位变成 30 次无脑点击。
- 开场白行尾显示字符数徽章，>200 字符标红（平台限制 200 字符，提前暴露）。
- 提示词行的复制确保代码块内原文（不含 ``` 围栏本身）。
- 「已复制」状态持久化到任务数据（重启工作台不丢）。

## 3. 功能三（P1 预研）：平台表单注入勘测

> 本期不做全自动填写，只做勘测与机制铺垫。原因：公司平台表单 DOM 结构未知，盲做注入必返工。

- 架构师产出**勘测报告**：在公司平台真实「卡片编辑」页面上，用 DevTools 采集五个字段输入框 + 总任务字段输入框的稳定选择器（id/name/label 邻接关系），评估是否存在富文本编辑器（contenteditable/draft-js 类，决定注入方式）。
- 机制铺垫：工作台偏好新增 `platformFieldMap`（选择器 → 字段名映射，JSON 可编辑），并实现**单字段试注入**命令（对当前激活 webview 按映射注入一个字段，验证 CDP `DOM`/`Input` 路径可行）。
- 验收口径：勘测报告 + 映射配置可保存 + 任一文本字段试注入成功即算 P1 达标；全自动整卡填写依据勘测结论列入 V3.5。

---

## 4. 范围外（明确不做）

- 自动点击平台「提交/保存」按钮（提交动作永远由用户人工确认）
- 豆包封面图自动生成/自动上传（图片流转走 V3.1 文件总线既有能力）
- 评价标准的结构化拆分（整段复制即可）
- Hermes 会话内容自动抓取（以 cards.md 文件为唯一入口，避免依赖 Hermes 页面 DOM）

---

## 5. 验收标准

1. 用一份真实 Hermes 产出（≥4 卡，含评价标准与测试人格）保存为 cards.md → 全部卡片与字段解析正确；故意打乱一段格式 → 落 unparsed 区展示，其余卡片不受影响。
2. 卡片舱：逐字段复制内容与原文逐字节一致（提示词不含围栏、保留内部换行）；已复制状态打勾、持久化、重启不丢。
3. 流式模式：开启后按顺序高亮，复制后自动推进到下一个未复制字段，全部完成有完成提示。
4. 开场白超 200 字符标红徽章正确。
5. P1：勘测报告交付；platformFieldMap 可配置保存；公司平台页面任一字段试注入成功。
6. 零回归：`npm run check` 通过；V3.1/V3.2/V3.3 全部既有功能按回归清单抽测通过；`parseCardsDocument` 配套单测（参照 parse-todo-lines.test.js 风格）。
