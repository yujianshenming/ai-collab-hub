一个面向 Windows 的 Electron 桌面工作台。把你所有的工作入口——常驻网页、企业微信、Codex / Cursor、终端、文件夹——全部集成进一个应用，并用「任务驱动 + 文件总线 + 卡片舱」打通能力训练搭建的端到端流程。

Overview

这是什么

用户的真实工作是「能力训练搭建」：从企业微信接任务、用 Hermes 分阶段写提示词、填到公司平台、生成封面图、浏览器插件测试、上传评估、循环修改直到通过。整套流程涉及十几个网页、多个文件夹、反复的下载/上传/复制粘贴。工作台的目标就是把这条链路收进一个桌面应用里。

常驻浏览器壳

10+ 网页以常驻 webview 标签承载，切换不重载、保留登录态、表单与滚动位置。支持地址栏、左右/上下分屏、Chrome 扩展加载。

任务驱动中枢

任务中心做落地页，每个任务有专属文件夹与五步流水线，自动管理产物、下载与上传，状态可回写到「待做任务.txt」。

内置终端

底部可拖拽高度的本地 PowerShell 终端（node-pty + xterm），并支持把 CLI 工具（如 Claude Code）作为独立标签常驻运行。

End-to-end Workflow

它在替你跑的 10 步工作流

这是工作台所有功能的设计依据——每一个模块都对应着这条链路上原本需要手工完成的环节。

企业微信查看本周任务手动整合到桌面「待做任务.txt」
按任务找人拿任务文档放入对应文件夹
打开固定页面 + 启动 Hermes常驻 webview，登录态不丢
任务文档发给 Hermes分阶段 → 审核 → 逐阶段写提示词
产出逐字段填到公司平台卡片名称 / 建议轮次 / 阶段描述 / 开场白 / 提示词
豆包生成封面图裁切底部约 100px 去水印 → 填到平台
填写评价标准总任务级字段
插件测试 → 下载测试对话与任务文档一起上传评估页
不通过 → 下载评估报告连同测试对话给 Hermes 修改 → 循环
通过 → 提示词保存归位任务完成

Diagnosis

为什么会觉得「乱、不知道有什么功能」

根因不在代码，而在 文档与代码严重脱节。三处自相矛盾，让人一打开文档就被带偏：

文件文档里写的实际代码问题
status/current.md停在 2026-06-09，目标还是「菜单栏重构」已经做到 V3.4落后 5 个版本
roadmap.mdV3.4「待派发」、V3.3「验证中」V3.4 已提交完成状态未更新
workbench/README.md只讲「网页标签 + 终端」早已是流水线 + 文件总线 + 卡片舱完全过时

建议下一步先把这三份文档对齐到 V3.4 真实状态——成本低、立刻止乱，以后随时打开就知道「有什么功能、到哪了」。

Known Issues

findTabByUrlPart 对无 url 字段的标签（desktop-app / cli-app）抛 TypeError。Hermes 阶段或点「加载至 Hermes」时即崩。 renderer.js:1938
