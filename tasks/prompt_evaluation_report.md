# Prompt and Dialogue Evaluation Report

## 1. Executive Summary (执行摘要)
本报告针对 HermesAgent 自动生成的系统提示词（System Prompt）及模拟对话录（Simulation Transcript），与用户手写完成的提示词及对话设计进行了多维度的深度对比与评估。评估覆盖了《物理治疗评定》（脑卒中后上肢功能评估，被动模式）和《运动控制系统》（巡检机器人驱动控制，导师模式）两个典型任务。

**主要结论：**
* **生成约束（Constraints）**：HermesAgent 在生成约束的执行上表现极佳，成功过滤了动作神态描写（无括号/星号噪音）、限制了回复字数（<100字）、消除了思维链（<think>）输出，并精确执行了跳转词切档。
* **结构与完备性（Structure & Rigor）**：用户手写的提示词采用分阶段（Card-by-Card）多文件结构，包含极高精度的领域参数（如临床肌力分级、ASR限幅与量化误差公式）。相比之下，HermesAgent 生成的提示词受限于 300 字的系统提示词限制，将多阶段高度压缩，丢失了具体的领域数据，导致 AI 模拟出的病情/工程参数流于表面。
* **对话模式（Dialogue Modes）**：导师模式（Tutor Mode）下启发式提问的对齐度很高；但在被动模式（Passive Mode）下，AI 患者存在首轮主动提问的逻辑漏洞，违背了被动应答的基本原则。

---

## 2. Prompt Structure and Rigor (提示词结构与完备性对比)

### 用户手写提示词
* **结构设计**：采用分阶段（Card-by-Card）的独立提示词设计。每一阶段包含 `## Role`, `## Context & Task`, `## Workflow`, `## Jump Condition`, `## Response Constraints` 等清晰板块，结构规范且易于大模型理解。
* **数据完备度**：**极高**。
  * **脑卒中任务**：详尽规定了 7 类客观检查的临床表现（例如：三角肌外展肌力2级、肘屈肌力3级、改良 Ashworth 1级、指鼻试验睁眼/闭眼表现等）。
  * **运动控制任务**：给出了精确的采样周期（1ms）、编码器参数（1040线×4倍频）、量化误差计算公式（`Δn = 60 / (1040×4×0.001) ≈ 14.4 rpm`）以及滤波参数。

### HermesAgent 生成提示词
* **结构设计**：将所有阶段高度压缩至一个**单一系统提示词**（System Prompt）中，且受制于 300 字的总长度约束。
* **数据完备度**：**较低**。生成的提示词仅概括了各阶段的名称和基本任务（例如：“阶段一确认前置...阶段二只查ASR是否饱和...”），丢失了所有具体的临床检查指标和工程计算公式。

### 对比结论
用户手写提示词在专业深度与数据完备性上显著优于 HermesAgent。HermesAgent 将所有卡片压缩在单条 System Prompt 中，导致大模型在仿真时缺乏具体的数据支撑，只能通过自身的常识“泛泛而谈”，极大降低了实训仿真的专业可信度。

## 3. Dialogue Mode Alignment (对话模式对齐分析)

### 3.1 被动角色扮演模式 (Passive Mode - 脑卒中患者李阿姨)
* **模式要求**：学生主导对话，AI 采取被动回应原则，不主动发问，不延伸话题。
* **问题暴露**：在 HermesAgent 的模拟对话中，AI 患者（Trainer）的第一句台词是：
  > `[Trainer] 治疗前我先了解情况，您右手现在最困扰什么？`
  这严重违反了被动应答原则——患者主动发问并引导学生开始评估。
* **原因分析**：这是由于 `AgentSandbox.simulate()` 默认强制输出第一张卡片的 `opening` 作为首轮对话，而 `TaskAnalyzer` 生成 of `opening` 为提问句式。对于被动模式，首轮对话应由学生（Student）发起，AI 患者应保持静默或仅作被动首句（如“医生您好，您帮我看看”）。

### 3.2 导师引导模式 (Tutor Mode - 专家严工)
* **模式要求**：Socratic 启发式引导，一次只问一个具体问题，逐步追问，不直接给答案。
* **执行情况**：**表现优异**。AI 导师在对话中逐一引导学生观察振荡频率、转速波形，并计算量化误差。
  > `[Trainer] 先只说现象里的抖动频率。你观察到振荡大概多快，是否接近固定频率？`
  > `[Trainer] 再只说转速波形。它是正弦摆动、锯齿、还是不规则抖动？幅值大概多大？`
* **差异点**：手写提示词在各阶段使用不同的切档跳转词（如 `准备就绪`、`现象确认`、`根源定位`），而 HermesAgent 统一使用全局跳转词（如 `下一阶段`），在多阶段独立卡片系统中的灵活性稍显不足。

## 5. Simulation Transcript Quality (模拟对话录质量评估)

### 5.1 脑卒中任务（被动模式）
由于缺乏具体的肌力、肌张力和关节活动度参数，模拟对话的真实度大打折扣。
* **学生表现**：学生一口气罗列了多项检查指令（“先看姿势和肿胀。再摸肌张力和疼痛。然后查关节活动度。再测肌力。”），这在真实临床问诊中是不合常理的。
* **患者回应**：AI 患者只能泛泛回答：“摸着右边有点紧，抬高时肩这儿有点疼。活动没左边大，力气也差些。” 缺乏具体的临床测量数据（如 ROM 角度、肌力分级），无法支持学生做出精准 of SOAP 评估和康复计划。

### 5.2 运动控制任务（导师模式）
启发式引导非常成功，对话非常贴合真实的远程协同调试场景。
* **计算引导**：大模型巧妙地避开了复杂标点，将公式用文字表述：
  > `[Trainer] 可用 Δn等于1除以1040乘4再乘60再除以0点001 结果大约多少rpm`
  这种处理虽然略显口语化，但在严格标点约束下成功引导了学生完成量化计算（`大概14到15 rpm`），具有极强的引导价值。

## 6. Recommendations for Improvement (核心优化建议)

1. **取消单条 System Prompt 的 300 字字数限制**：
   目前的 PromptGenerator 为了追求 Concise，将 System Prompt 限制在 300 字内。这导致大量关键参数被丢弃。应当允许 System Prompt 包含从任务文档中抽取的关键数据（如临床指标、数学公式）。
2. **支持 Card-by-Card 动态编译 Prompt**：
   不应该只使用单条全局 Prompt 驱动整个仿真。在 `AgentSandbox.simulate` 中，应随卡片切换动态更新 System Message（使用 `compile_card_prompt` 生成的卡片提示词），让 AI 能够在不同阶段读取到当前阶段专属的症状库或参数公式。
3. **修复被动模式首轮主动发问的 bug**：
   对于被动角色扮演模式，检测并重写第一张卡片的 `opening`，或允许由模拟学生（Student）率先发起会话，避免 AI 扮演的患者主动掌控话题。
4. **动态跳转词适配**：
   在卡片编译时，应使用 task_document 提取到的多阶段差异化跳转词（如 `准备就绪`、`现象确认`），而不是强制全局统一使用单一跳转词。
