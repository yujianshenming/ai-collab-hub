## 0. 背景与目标

用户每周从企业微信收任务后，手动整合到桌面的「待做任务.txt」，再到工作台里手动逐条录入任务卡片 —— 两边重复维护。

**目标：工作台一键解析待做任务.txt 导入为任务卡片**，并补齐用户真实工作流缺少的两块状态模型：「未提交」中间态和子任务粒度进度。

真实文件样例（用户截图实录）：

## 1. 功能一：待做任务.txt 解析与导入

### 1.1 行解析规则

每行模式：`学校《课程》 [任务类型] [N个] [状态(可带括号备注)] [负责人] [星期]`

- **锚点**：`学校` = 行首到 `《` 之前；`课程` = 《》内文字。无书名号的行跳过（计入"无法解析"清单）。
- **任务类型**：在《》之后匹配关键词：`能力训练搭建 / 能力训练修改 / 能力训练验收 / 作业批阅搭建 / 作业批阅验收 / 作业批阅修改`。映射到现有类型枚举；`作业批阅修改` 现有枚举没有 → 类型枚举补充该项。匹配不到 → 类型留空（导入预览中标黄提醒）。
- **数量**：`(\d+)个`，缺省为 1。
- **状态**：`已完成 / 未完成 / 未提交`，可带全角/半角括号备注。缺省为 `未完成`。映射：未完成→pending、未提交→unsubmitted（见功能二）、已完成→completed。
- **括号备注**：形如 `（已完成任务3/4/5/6，任务2待确认）`，解析出子任务编号与状态（见功能二）；解析失败时整段存入任务的 `note` 字段原样展示，不丢信息。
- **负责人**：状态之后、星期之前的非空 token（可能含数字，如 `李漫1`）。
- **星期**：`周一`~`周日`，缺省为空。
- 字段间空格数量不定（样例里有单空格、双空格），解析需容错。

### 1.2 导入交互

- 任务中心页头新增次级按钮「从待做任务导入」（添加任务旁，ghost 样式）。
- 首次使用：弹文件选择（系统对话框）定位 txt；选定后路径存入工作台偏好（workbench-prefs.json，沿用 V3.1 偏好机制），之后一键直读，偏好面板可改路径。
- 点击后弹**导入预览 dialog**（居中 modal，复用现有 dialog 风格）：
  - 三个分组列出解析结果：**新增**（工作台没有的）/ **更新**（已存在但字段有变化的，逐字段显示 旧→新）/ **无变化**（默认折叠）。
  - 匹配键：`学校 + 课程 + 任务类型` 三元组。
  - 无法解析的行单独一组原文展示，不阻塞导入。
  - 每条可单独勾选/取消（默认：新增全选、更新全选、无变化不选）。
  - 底部「导入所选」/「取消」。导入后 toast 汇总（新增 x 条、更新 y 条）。
- **单向导入**：只从 txt → 工作台。不写回 txt（范围外，见 §3）。
- 更新规则：只覆盖 quantity/status/owner/weekday/subtasks/note 字段；**绝不触碰** taskFolder/chatLogPath/reportPath 等产物字段。

### 1.3 实现要点

- 新 IPC `tasks:read-todo-file`：main 读偏好里的 txt 路径（UTF-8，兼容 BOM；若乱码尝试 GBK 回退——用 Buffer + 简单启发：UTF-8 解码出现替换字符则按 GBK 用 iconv？**不引新依赖**，Node 原生不支持 GBK，方案改为：先按 UTF-8 读，若含 U+FFFD 替换字符则提示用户"文件编码非 UTF-8，请用记事本另存为 UTF-8"，不做转码）。返回原文，**解析逻辑放 renderer**（纯函数 `parseTodoLines(text)`，便于后续测试工程师写用例）。
- 该 IPC 只允许读偏好中登记的这一个文件路径，preload 不暴露通用读文件能力。

## 2. 功能二：「未提交」状态 + 子任务进度

### 2.1 未提交状态（unsubmitted）

用户语义：**测试通过了，但还没提交到公司平台**。介于"做完了"和"交付了"之间。

- 状态枚举新增 `unsubmitted: "未提交"`；徽章配色用 info 蓝（`--info` / `--info-bg`）。
- 任务中心分组调整：`待处理 / 已暂停` → `未提交` → `已完成` 三段（未提交组排在中间，无任务时该组不渲染）。
- 未提交卡片主按钮：**「标记已完成」**（teal）——用户在平台提交后手动点击 → completed。「⋯」菜单含「重新打开任务」（→pending）与编辑/删除。
- 状态流转入口：进行中任务流程走完（Hermes 诊断后/手动结束）时，结束确认从直接 completed 改为询问「任务已完成测试，是否已提交平台？」：已提交→completed / 未提交→unsubmitted。统计行的"已完成"卡改为"未提交 + 已完成"两个数字合并显示或拆分（统计行从 4 卡变 5 卡放不下时，把"本周任务"和"进行中"合并为一张，保持 4 卡）。

### 2.2 子任务进度

- 任务模型新增 `subtasks: [{ index: number, status: "pending"|"done"|"unconfirmed" }]`，长度与 quantity 联动（quantity 变化时尾部增删）。
- 来源一：导入时解析括号备注（`已完成任务3/4/5/6` → 对应 index 标 done；`任务2待确认` → unconfirmed）。
- 来源二：手动维护——任务卡片上的进度条区域改为可点击，点开**子任务清单 popover**：N 行「任务 1..N」，每行三态切换（待做/已完成/待确认）。
- 卡片进度条与文案升级：流水线进度（执行中任务）保持现状；非执行中任务若有 subtasks，进度条显示 `子任务 x/N 完成`（x=done 数量），全 done 时提示可标记未提交/已完成。
- `note` 字段（解析失败的备注原文）显示在卡片副标题下方小字。

### 2.3 星期字段

- 任务模型新增 `weekday`（"周一"~"周日"或空）。卡片元信息行追加显示（如 `周三交付`）。
- 待处理分组内排序：有 weekday 的按周一→周日，无的排最后；同星期按创建顺序。

## 3. 范围外（明确不做）

- 写回/反向同步待做任务.txt
- 企业微信直接抓取任务
- 定时自动导入（手动按钮触发即可）
- GBK 自动转码（提示用户转存 UTF-8）
- 子任务级别的独立流水线/文件夹

---

## 4. 验收标准

1. 用 §0 的 8 行样例实测：8 行全部解析成功，字段逐项正确（含 `李漫1` 这种带数字负责人、双空格行、无类型行、无状态行）；故意加一行无书名号文本 → 进"无法解析"组不阻塞。
2. 导入预览：新增/更新/无变化分组正确；更新组逐字段 diff 展示；二次导入同一文件 → 全部落在"无变化"。
3. 更新导入不触碰产物字段：对一个有 taskFolder/chatLogPath 的既有任务改 txt 中数量后导入 → 仅 quantity 变化，产物字段与文件夹原样。
4. 未提交状态：分组/徽章/「标记已完成」/结束任务时的询问分支全部生效；重新打开任务对 unsubmitted 卡同样可用。
5. 子任务：括号备注正确映射（3/4/5/6 done、2 unconfirmed、1 pending）；手动三态切换并持久化；进度条文案随之变化。
6. 既有功能零回归：流水线/文件总线/分屏/暂停恢复不受影响；`npm run check` 通过；新 IPC 仅能读偏好登记的单一 txt 路径。

### 提交要求
- 功能一（解析+导入）一个 commit，功能二（状态+子任务+星期）一个 commit，各自过 `npm run check`。
- `parseTodoLines` 写成无副作用纯函数并导出（或挂 window 便于测试注入），方便测试工程师写解析用例。
- 完成后向产品经理回报：改动清单 + 六条验收自检。
